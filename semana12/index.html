Semana 12: Desarrollo Backend – Jakarta EE y JSP

Descripción de temas aprendidos

En esta semana, profundizamos en el desarrollo backend utilizando Jakarta EE, la evolución moderna de Java EE (Enterprise Edition), impulsada por la Fundación Eclipse. Aprendimos que Jakarta EE es un conjunto de especificaciones para construir aplicaciones empresariales robustas, seguras y escalables en el ecosistema Java.

Uno de los componentes clave que estudiamos fue Jakarta Server Pages (JSP), una tecnología que permite crear páginas web dinámicas al incrustar código Java dentro de archivos HTML. Aunque su uso ha disminuido frente a frameworks modernos como Spring, JSP sigue siendo relevante para entender cómo funciona el backend en Java.

El flujo de procesamiento de una página JSP es el siguiente:

  1. El usuario solicita una página JSP (ej: index.jsp).
  2. El servidor (como Tomcat) intercepta la solicitud.
  3. El contenedor JSP traduce el archivo JSP a un Servlet Java.
  4. El servlet es compilado a bytecode (*.class).
  5. El servlet se ejecuta, generando una respuesta en HTML.
  6. El HTML resultante se envía al navegador del cliente.

Para que este proceso funcione, necesitamos un contenedor de servlets, como Apache Tomcat, que implementa las especificaciones de Jakarta EE.

Elementos clave de JSP

Los elementos principales que permiten la dinámica en una página JSP son:

  • Scriptlets (<% ... %>): Bloques de código Java que se ejecutan en el servidor.
  • Expresiones (<%= ... %>): Muestran el valor de una expresión Java.
  • Declaraciones (<%! ... %>): Definen variables o métodos a nivel de clase.
  • Directivas (<%@ ... %>): Controlan el comportamiento de la página (ej: page, include, taglib).
  • Acciones (): Elementos XML que realizan tareas como incluir archivos o redirigir.

Objetos implícitos en JSP

JSP proporciona 9 objetos implícitos que facilitan la manipulación de datos sin necesidad de instanciarlos:

  • request: Representa la solicitud HTTP del cliente.
  • response: Permite enviar la respuesta al cliente.
  • out: Usado para escribir contenido en la respuesta.
  • session: Almacena datos específicos de la sesión del usuario.
  • application: Objeto global para toda la aplicación web.
  • config: Configuración del servlet.
  • page: Referencia al objeto JSP actual.
  • pageContext: Acceso a todos los objetos de contexto.
  • exception: Disponible en páginas de error.

Ejercicios de laboratorio – Resultados

Realizamos los siguientes ejercicios prácticos:

  • Ejercicio 1: Instalación de JDK 19, IntelliJ IDEA y Apache Tomcat.

    Seguimos el procedimiento para instalar:

    • OpenJDK 19: Desde https://jdk.java.net/19/, configurando las variables de entorno JAVA_HOME y PATH.
    • IntelliJ IDEA Community Edition: IDE para desarrollar aplicaciones Java.
    • Apache Tomcat 10: Descargado desde https://tomcat.apache.org/, instalado y configurado para ejecutar en el puerto 8080.

    Verificamos el funcionamiento accediendo a http://localhost:8080 y visualizando la página de bienvenida de Tomcat.

  • Ejercicio 2: Crear un proyecto web en IntelliJ y desplegarlo en Tomcat.

    En IntelliJ IDEA, creamos un nuevo proyecto Maven con arquetipo webapp. Configuramos el pom.xml con las dependencias necesarias y agregamos un archivo index.jsp en src/main/webapp.

    Luego, configuramos Tomcat como servidor de aplicación en IntelliJ y desplegamos el proyecto. Al ejecutarlo, se abrió el navegador mostrando nuestra página JSP con la fecha actual.

  • Ejercicio 3: Crear una aplicación JSP con formulario y contador de visitas.

    Desarrollamos una página que permitía al usuario ingresar su nombre y mostrar un saludo personalizado. Además, usamos el objeto session para contar cuántas veces el usuario había accedido a la página en esa sesión.

    Integer contador = (Integer) session.getAttribute("contador");
if (contador == null) contador = 0;
session.setAttribute("contador", ++contador);
out.println("
    Visitas en esta sesión: " + contador + "
    ");

    La aplicación funcionó correctamente, demostrando el uso de objetos implícitos y la persistencia de datos en la sesión.

Reflexión: ¿Qué aprendí? y ¿Cómo aprendí?

Esta semana fue fundamental para entender los cimientos del desarrollo backend en Java. Aprendí que JSP, aunque es una tecnología antigua, sigue siendo un excelente ejemplo de cómo se integró la lógica del servidor con la presentación en la era temprana de la web. Me impresionó ver cómo una página JSP se convierte en un servlet y se ejecuta en Tomcat, lo que me dio una visión más profunda del ciclo de vida de una aplicación web Java. Aprendí haciendo: cada error en la configuración de Tomcat o en la sintaxis de JSP fue una oportunidad para investigar, leer la documentación y reforzar mi comprensión. Sentí una gran satisfacción al ver mi primera aplicación JSP funcionando en el servidor. Ahora tengo una base sólida para avanzar hacia frameworks más modernos como Spring.
